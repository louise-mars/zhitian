# 知天 — 完整产品文档

> 天气 × 日程 × 文化，三位一体的智能生活助手

---

## 一、产品定位与差异化

### 为什么世界需要另一个天气 App

全球应用商店有超过 200 个天气应用。用户不缺天气数据——他们缺的是**基于天气的决策能力**。

| 当前天气 App 的问题 | 本质 |
|------|------|
| 告诉你"明天下雨"，但不告诉你该怎么办 | 信息 ≠ 决策 |
| 天气和日程是两个独立的 App | 用户需要手动关联 |
| 界面千篇一律，打开即关 | 没有情感连接，没有留存理由 |
| 广告满屏，隐私泄露 | 用户是产品，不是客户 |

**知天的回答：把天气从"被动查询工具"变成"主动决策助手"，同时用文化内容建立情感连接。**

### 一句话定位

> **知天是一个将天气预报、日程管理与中国传统文化融为一体的智能生活助手。**

### 三层价值模型

```
┌─────────────────────────────────────────┐
│  第三层：情感价值（文化滋养）              │
│  每日诗词 · 农历节气 · 东方美学视觉        │
├─────────────────────────────────────────┤
│  第二层：决策价值（主动助手）              │
│  日程天气预警 · 智能建议 · 主动推送        │
├─────────────────────────────────────────┤
│  第一层：工具价值（精准数据）              │
│  15天预报 · 分钟级降雨 · 生活指数          │
└─────────────────────────────────────────┘
```

大多数天气 App 只做到第一层。知天的护城河在第二层和第三层。

### 核心差异化

**1. 日程天气预警引擎（AlertEngine）**— 市场唯一

将用户日程（App 内 + 系统日历）× 15 天天气预报 × 气象灾害预警三者交叉匹配，生成智能预警。具备室内事件过滤、小时级精确匹配、降水量阈值、灾害预警关联、本地推送通知（未来 3 天）。

**2. 文化内容层（诗词系统）**— 情感粘性

400+ 首唐诗宋词按天气/季节/节气智能匹配，一周不重复。桌面 Widget 诗词模式，不打开 App 也能每天看到一句诗。

**3. 东方内敛美学**— 视觉差异化

宫崎骏风格天气动画、64dp Canvas 图标微动画、700ms 背景转场、低电量自动降级。轻量、克制、高效。

## 二、系统架构设计

### 整体架构

```
┌─────────────────────────────────────────────────────────┐
│                        UI Layer                          │
│  HomeScreen · CalendarScreen · SettingsScreen · Widgets  │
├─────────────────────────────────────────────────────────┤
│                     ViewModel Layer                      │
│  HomeViewModel · CalendarViewModel · CityViewModel      │
├─────────────────────────────────────────────────────────┤
│                     Domain Layer                         │
│  AlertEngine · SuggestionGenerator · DailyPoetry        │
│  AnimationDegradationManager · LunarCalendar            │
├─────────────────────────────────────────────────────────┤
│                      Data Layer                          │
│  QWeatherRepository · WeatherRepository · EventRepo     │
│  CalendarRepository · CityRepository · UserPrefsRepo    │
├─────────────────────────────────────────────────────────┤
│                    Infrastructure                        │
│  Room DB · DataStore · QWeather API · Amap SDK          │
│  AlarmManager · NotificationManager · Glance Widget     │
└─────────────────────────────────────────────────────────┘
```

### 数据流（首页加载）

```
App 打开
  → HomeViewModel.loadData()
    → 阶段一（零网络）：SharedPreferences 缓存坐标 → Room 缓存天气 → 立即渲染
    → 阶段二（网络）：高德 SDK 定位 → QWeather API → 渲染新数据
      → AlertEngine.generateAlerts(daily, hourly, warnings)
        → 查询 EventRepository + CalendarRepository
        → 室内过滤 → 天气匹配 → 灾害关联
      → WeatherAlertNotifier.checkAndNotify() → OS 通知
      → WidgetRefreshWorker → 更新桌面 Widget
```

### 预警系统架构

```
┌─────────────────────────────────────────┐
│           AlertEngine                    │
│                                          │
│  输入：                                   │
│  ├── dailyForecasts (15天)               │
│  ├── hourlyForecasts (24小时)            │
│  └── warnings (气象局预警)               │
│                                          │
│  处理：                                   │
│  ├── 查询事件（App + 系统日历）           │
│  ├── 室内关键词过滤（20+ 词）            │
│  ├── 天气匹配                            │
│  │   ├── 今天定时事件 → ±1小时精确匹配   │
│  │   └── 未来事件 → 日级匹配             │
│  ├── 灾害预警关联（今明两天）            │
│  └── 去重 + 排序 + 截断（≤5条）          │
│                                          │
│  输出：                                   │
│  ├── 首页卡片（ScheduleWeatherAlertCard）│
│  └── OS 通知（未来3天，每天≤1条）        │
└─────────────────────────────────────────┘
```

### 动画系统架构

```
AnimationDegradationManager
  ├── 检测：BatteryManager + PowerManager（App resume 时）
  ├── 状态：StateFlow<DegradationState>
  │   ├── 正常：全动画 + 700ms 转场
  │   ├── 低电量(<15%)：静态图标 + 200ms 转场
  │   └── 省电模式：静态图标 + 即时切换
  └── 滞后：15%禁用 / 20%恢复（避免频繁切换）

WeatherIconAnimator (64dp Canvas)
  ├── 太阳：光芒脉冲（3-5s 周期）
  ├── 云：水平漂移（≤20dp/s）
  ├── 雨：3-5 滴下落
  ├── 雪：3-5 片飘落+横摆
  ├── 雷暴：闪电闪烁（3-7s 间隔）
  └── 雾：3 层薄雾流动

WeatherAnimationOverlay (全屏粒子)
  ├── 宫崎骏风格：丁达尔光束、手绘雨滴、萤火虫
  └── 季节花瓣/落叶（春粉/夏绿/秋金）
```

### 首页布局层级

```
HeaderBar（城市名 + 日期 + 农历）
  ↓
CurrentWeatherCard（温度 + 天气图标微动画）
  ↓
SmartAdviceCard（一句话智能建议）
  ↓
每日诗词（暖金色衬线字体）
  ↓
🚨 气象灾害预警（红/橙/黄/蓝色边框，最醒目位置）
⚠️ 日程天气预警（日程×天气冲突提醒）
  ↓
小时预报（横向滚动）
今日日程
  ↓
降雨预报
15天预报（折线图 + 详情列表）
生活指数
```

## 三、设计过程中的挑战与经验教训

### 中国大陆适配挑战

1. **GPS 定位不可用**：Google Play Services 在中国不可达 → 集成高德定位 SDK
2. **海外 API 不可达**：Open-Meteo/Nominatim 全部超时 → 改用和风天气
3. **和风天气域名变更**：2026 年改为项目专属域名 → 从控制台获取
4. **免费版 API 限制**：AQI 返回 403 → 优雅降级（隐藏卡片）

### 预警系统设计挑战

5. **误报控制**：初版对所有事件都预警 → 三层过滤（室内/阈值/小时级）
6. **精确率 > 覆盖率**：一条误报比十条漏报更伤害用户信任
7. **预警位置**：最初放在第二屏 → 提升到诗词下方第一位（最醒目）
8. **通知频率**：每天最多 1 条，覆盖未来 3 天

### 性能与体验挑战

9. **动画性能预算**：全屏粒子 + 图标动画 + 转场 → 自动降级管理
10. **深色/浅色模式**：天气渐变背景覆盖 Material 主题 → 独立浅色渐变色系
11. **温度图表可见性**：浅蓝线在蓝色背景上不可见 → 金黄+淡紫高对比度
12. **分享功能**：系统分享面板需要正确的 Intent flags 才能显示社交 App

### 可靠性挑战

13. **Room 多实例**：Widget/Worker 各自建 DB → 统一 migration 链
14. **重复事件月末**：1月31日月重复在2月消失 → `coerceAtMost(lastDayOfMonth)`
15. **AlarmManager vs WorkManager**：精确提醒需要 AlarmManager + BOOT_COMPLETED
16. **定位 fallback**：GPS → 缓存 → 默认城市，永不卡死

### 关键设计原则

- **信息层级**：紧急信息（预警）> 决策信息（日程冲突）> 参考信息（预报）
- **优雅降级**：功能不可用时隐藏，不显示错误状态
- **用户逃生路径**：任何状态都保留导航入口
- **性能尊重**：省电模式 = 用户明确表达"性能优先"，必须尊重

## 五、使用指南

### 安装与配置

1. 从 GitHub Releases 下载 APK 或自行编译
2. 首次打开授予定位权限（可跳过，使用默认城市）
3. 授予日历读取权限（可选，用于日程天气预警）
4. 授予通知权限（可选，用于预警推送和事件提醒）

### 首页功能

- **下拉刷新**：强制更新天气数据
- **左右滑动**：切换今天/明天/后天天气
- **点击城市名**：打开城市选择器
- **点击诗词**：展开完整全文
- **点击分享图标**：生成天气卡片图片分享到微信/微博等

### 日程天气预警

- 自动运行，无需配置
- 在首页诗词下方显示预警卡片
- 线上会议等室内事件自动过滤
- 未来 3 天有预警时会收到系统通知

### 日历功能

- **点击日期**：查看详情 + 添加/编辑事件
- **左右滑动**：切换月份
- **添加事件**：标题 + 描述 + 时间 + 提醒 + 颜色 + 重复规则
- **提醒选项**：开始时 / 5分钟前 / 15分钟前 / ... / 1周前

### 桌面 Widget

- 长按桌面 → 添加小组件 → 选择"知天"（天气）或"知天·诗词"（诗词）
- 天气 Widget：每小时自动刷新
- 诗词 Widget：每日自动更换

### 设置

- **温度单位**：摄氏/华氏
- **主题模式**：跟随系统 / 浅色 / 深色
- **默认城市**：定位不可用时的 fallback
- **天气通知**：天气变化提醒开关

## 六、技术护城河

| 能力 | 竞品复制成本 | 原因 |
|------|------------|------|
| 日程天气预警引擎 | 高 | 需要重新设计产品架构，不是"加个模块"能实现的 |
| 400+ 诗词智能匹配 | 高 | 需要文学素养 + 工程结合 + 内容积累 |
| 灾害预警×日程关联 | 高 | 需要打通多数据源 + 智能过滤逻辑 |
| 东方美学动画系统 | 中 | 设计能力依赖 |
| 中国区原生适配 | 中 | 需要高德 SDK + 和风天气集成经验 |

核心壁垒在**产品理念的一致性**——天气、日程、文化三者的深度融合需要从架构层面重新设计。

## 八、总结

**知天不是又一个天气 App。**

它是一个回答"我该怎么办"的决策助手，一个每天用一句诗开启新一天的文化伴侣，一个尊重用户隐私和设备性能的工具。

用户不缺天气数据。他们缺的是：
1. 有人帮他们把天气和日程关联起来，主动提醒
2. 一个每天值得打开的理由（不是广告，是诗词）
3. 一个不偷数据、不耗电、不烦人的工具

**知天，知时节，知冷暖，知进退。**
